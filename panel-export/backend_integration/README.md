# Интеграция веб-панели с ботом (backend)

Эта папка — не часть фронтенда. Это Python-модуль, который нужно **скопировать
внутрь исходников самого бота** (`/tmp/botcode` в нашей переписке — у вас это
корень проекта бота), чтобы веб-панель могла читать и менять настройки чатов
через настоящий REST API, работающий поверх тех же данных, что использует бот.

## Что внутри

```
bot/webapi/
├── __init__.py
├── auth.py       — проверка Telegram initData / Login Widget, сессии
├── serializers.py— преобразование внутренних dict'ов бота в JSON для панели
├── providers.py  — каталог AI-провайдеров и моделей (как в chat_ai_router.py)
└── server.py     — aiohttp-приложение со всеми маршрутами + CORS
main_patch.py     — пример того, как запустить веб-сервер рядом с поллингом бота
```

## Установка

1. Скопируйте папку `bot/webapi/` в ваш проект бота (`<ваш_бот>/bot/webapi/`).
2. Установите зависимости (если их ещё нет): `pip install aiohttp aiohttp-cors PyJWT`.
3. Откройте `main_patch.py` — это не файл для копирования "как есть", а
   ориентир: перенесите вызов `start_webapi_server(...)` в ваш реальный
   entrypoint (файл, где вы сейчас запускаете `dp.start_polling(...)` или
   аналог), рядом с запуском поллинга, через `asyncio.gather(...)`.
4. Задайте переменные окружения на сервере бота:
   - `WEBAPI_PORT` — порт REST API (например `8081`).
   - `WEBAPI_CORS_ORIGIN` — домен, с которого фронтенд будет делать запросы
     (например `https://your-panel.vercel.app`, для разработки можно `*`).
   - `WEBAPI_SESSION_SECRET` — случайная строка для подписи сессионных токенов
     (сгенерируйте один раз: `python -c "import secrets;print(secrets.token_hex(32))"`).
5. Убедитесь, что `BOT_TOKEN` уже доступен там, где создаётся `Bot(...)`
   (`bot/core/loader.py` в вашем проекте) — модуль `auth.py` берёт токен из
   `bot.token` того же экземпляра `aiogram.Bot`, который уже использует бот,
   отдельный секрет не нужен.

## Важные примечания

- **Токен бота, найденный в загруженном архиве, был в открытом виде в
  `bot/core/config.py`.** Он нигде не используется в этом коде и не
  копировался — но раз он утёк в архиве, который проходил через сторонние
  системы, настоятельно рекомендуется перевыпустить токен через @BotFather
  (`/revoke` → `/token`) и обновить его на сервере бота.
- Этот модуль **не подменяет** ни одну существующую логику модерации — он
  только читает и пишет те же самые структуры (`bot.storage.state.settings`,
  `get_anti_raid_settings`/`save_anti_raid_settings`,
  `bot.services.chat_ai_router.*`, `bot.storage.providers_storage.*`), которые
  уже использует сам бот, и сохраняет через `save_settings(chat_id)` — то есть
  изменения из панели применяются мгновенно, бот не нужно перезапускать.
- Список чатов админа собирается так: перебираются все чаты, за которые у бота
  есть сохранённые настройки (папки `group_settings/<chat_id>/`), и для
  каждого выполняется `bot.get_chat_member(chat_id, user_id)` — попадают в
  список только чаты, где пользователь `creator` или `administrator`. При
  большом числе чатов это может быть медленно; при необходимости добавьте
  кеширование (см. `TODO` в `server.py`).
- Некоторые внутренние имена функций в ваших обработчиках (`bot/handlers/*.py`)
  начинаются с `_` (например `_ensure_antispam_cfg`, `_get_cfg` в
  `anti_shlyukhobot.py`) — это соглашение об "внутреннем" использовании внутри
  модуля, но не жёсткое ограничение Python. `webapi/server.py` читает и
  сохраняет через `bot.storage.state.settings` напрямую и вызывает
  `save_settings(chat_id)` — так же, как это делают сами обработчики команд —
  вместо того чтобы полагаться на приватные хелперы, которые могут отличаться
  между версиями кода. Если у вас после интеграции появятся расхождения
  (например другое имя поля), обновите `bot/webapi/serializers.py` — это
  единственное место, которое знает про форму JSON.
- Статистика антишлюхобота (`detected`/`passedCaptcha`/`banned`) в вашем коде
  хранится только в памяти процесса (`_stats`, не сохраняется на диск) — после
  перезапуска бота она обнулится. Панель отображает то же самое поведение.

## Проверка

После запуска бота с веб-сервером проверьте:

```bash
curl https://<адрес-сервера-бота>:<WEBAPI_PORT>/api/healthz
# {"status":"ok"}
```

Затем откройте фронтенд (см. `DEPLOY_GUIDE.md` в корне архива) и укажите этот
адрес в `VITE_API_BASE_URL`.
